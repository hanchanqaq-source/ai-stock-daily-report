# Errors and Lessons

## Portable-M3.4：成对下载失败不能留半成品

- 固定规则：ZIP 与 `.sha256` 任一下载失败或超时，清理仅由本次下载创建的临时目录；不触及用户选择的文件、程序目录或用户数据目录。

## Portable-M3.3：校验文件也必须有下载上限

- 固定规则：对 ZIP 与 `.sha256` 均检查 Content-Length 和实际流量，单文件超过 1.5 GiB 或长度非法即拒绝并清理应用创建的临时目标文件。

## Portable-M3.2：HTTPS 不等于可信更新来源

- 固定规则：下载和每一次重定向均只接受 `github.com`、`objects.githubusercontent.com` 或 `release-assets.githubusercontent.com`；其他 HTTPS 地址也拒绝。

## Portable-M3.1：已下载不等于已授权替换

- 固定规则：下载成功只保留在本次运行的临时目录；“安装已下载更新包”仍须重新校验并显示独立系统确认。应用重启后不恢复下载包引用。

## Portable-M3：下载与替换必须是两次明确动作

- 风险：把 GitHub 下载成功直接视为安装成功，会使网络下载越过用户对程序替换的确认边界。
- 固定规则：只接受 Release 中一一对应的 Portable ZIP 与 `.sha256`，下载到系统临时目录后立即校验；通过仅表示文件可用，绝不自动解压、替换或重启。

## Portable-M2.2E：自动回退也必须让用户可见

- 风险：自动更新或自动回退若没有重启后的明确结果，用户只能从版本或异常现象猜测发生了什么，容易再次重复操作。
- 固定规则：新版本健康通过和旧版本自动回退均通过受控恢复点结果标记，在重启后用本机原生提示告知；提示不包含 ZIP 路径、密钥或用户数据内容。

## Portable-M2.2D：进程未退出不等于应用已健康

- 风险：Windows EXE 即使仍在运行，内部 backend 也可能尚未启动、已失败或无法通过本机健康检查；把“存活数秒”当作更新成功会留下不可用的新程序。
- 固定规则：退出后的 helper 必须等待新主进程在实际 `127.0.0.1` backend 健康检查成功后写入的受控恢复点回执。75 秒内没有回执或进程退出，都停止新进程、只回退程序文件并启动旧版本。

## Portable-M2.2C：确认后的 ZIP 仍须由退出后的 helper 复验

- 风险：用户确认后到应用退出前，原 ZIP 仍可能被替换；只信任确认前的内存校验结果会让实际解压内容脱离 SHA-256 与结构门禁。
- 固定规则：helper 在替换前再次计算 SHA-256、临时解压并检查中文 EXE、backend EXE 与 `data/config/logs/plugins`。只有全部通过才删除可替换程序文件；任何异常均用恢复点回退，且不触碰用户目录。

## Build E2：已有生产持久化链路时不能再新增第二套 SQLite

- 风险：即使新模块本身具备目录抽象、迁移和事务，如果生产前端/API 已经在使用另一套数据库，再建立 `stock_fund_quality.db` 和重复的用户、股票、基金表仍会形成两个数据真相来源。
- 固定处理：先全仓审计生产调用方，优先保留已经连接前端、API、备份、回收站和历史记录的 SQLAlchemy workspace 链路；移除未被生产调用的平行实现，不做静默双写或双向同步。
- 验证：临时正式链路数据库只允许出现 `workspace_*` 业务表和既有 `schema_migrations`，不得出现第二套 `users`、`stock_holdings`、`fund_holdings`、`schema_version` 或 `data_migrations`。

## Build E2：SQLite 不能和程序更新、真实数据或测试数据混用

- 风险：更新覆盖 `stock_analysis.db` 会丢失用户数据；测试复用正式数据库会污染真实持仓；前端直接打开数据库会绕过 localhost API 和 SQLAlchemy 会话边界。
- 固定处理：正式后端只使用 `DATABASE_PATH` 指向的一个 SQLite 文件；Windows 更新保留 DB/WAL/SHM；自动测试只设置临时 `DATABASE_PATH` 并重建数据库会话；前端只调用 localhost API。

## 持久化阶段 E3：基金自选不能伪装成基金持仓

- 风险：若基金自选复用 `workspace_fund_holdings`，就必须伪造金额或收益字段，还可能让后续组合分析把“只是关注”误当成“已经持有”。
- 固定处理：基金自选使用同一正式 SQLite 中独立的 `workspace_fund_watchlist_items` 表，仍按 `user_id` 隔离；加入或移出自选不得修改基金持仓。
- 验证：同一代码可以同时存在于自选和持仓；移出自选后持仓仍保留；同一用户自选代码唯一，不同用户可分别关注同一代码。

## 既有工作台备份能力：备份不能等同于复制整个本机数据库

- 风险：直接导出 SQLite 文件、`.env` 或应用目录，会把无关设置、日志，甚至密钥/凭证一并带走，既不利于迁移也破坏安全边界。
- 固定处理：备份只使用版本化 JSON 契约，白名单限定为工作台用户、股票快速持仓、基金快速持仓和基金自选；未知字段、未知用户、重复 ID、非有限数字和未确认覆盖一律拒绝。
- 恢复原则：覆盖导入和恢复前都先创建本机恢复点，预览操作不得写库；恢复点同样只能保存这些工作台数据。

## Build E2：乐观更新失败后必须回到服务端真相

- 风险：新增、删除、切换用户等操作若先更新 React，再遇到本机写库失败，会让界面和数据库长期不一致；旧请求晚返回也可能覆盖较新的用户状态。
- 固定处理：mutation 按发起顺序写入；成功后应用完整服务端状态或重新读取，失败后重新读取完整状态并显示可重试错误；初次加载和 mutation 使用请求序号，旧响应不得覆盖新状态。

## Build E4：删除不能绕过可恢复路径

- 固定处理：快速持仓删除先写入本机回收站，再删除原记录；恢复时校验当前用户和 ID 冲突，防止跨用户恢复或覆盖新记录。

- PR95-FIX-01：安全词上下文误伤。页面层、demo payload 层、文档层不能使用同一套粗暴关键词规则；个人工作台允许资金看板、买入观察、实时涨跌等直观词语；真正禁止的是强制交易、自动执行、收益承诺和真实敏感值泄露。
- PR95-FIX-02：页面展示层不要使用容易被误判为系统代办的状态词，统一改为“记录状态 / 观察记录”。个人工作台可以使用直接易懂的观察词；测试应区分内部字段名和页面展示文案，不应使用粗暴关键词断言。
- PR95-FIX-03：记录状态测试契约同步。`auto_execute` 可作为内部兼容字段存在，但页面展示应使用“记录状态 / 观察记录”；测试只检查页面展示词，不把内部字段名等同于页面承诺。

## PR #164–#167：AkShare 真实只读链路复盘

### PR167-FIX-01：单源 Manager 不等于完全脱离通用 Config

- 现象：`create_akshare_only_manager()` 虽然只传入了一个 `AkshareFetcher`，但 `AkshareFetcher()` 构造器仍调用 `get_config()`。
- 风险：通用 Config 初始化会读取其他 Provider、AI、通知、账户等环境配置；与“真实只读路径只接触公开行情开关和公开数据”的边界不一致。
- 正确处理：为 `AkshareFetcher.__init__()` 增加 keyword-only 参数 `enable_eastmoney_patch: Optional[bool] = None`。
  - 显式 `False`：不导入、不调用 `src.config.get_config`，也不导入、不执行东财补丁。
  - 显式 `True`：直接执行东财补丁，但不读取通用 Config。
  - `None`：保留旧程序通过通用 Config 判断是否启用补丁的行为。
- 固定规则：Core-M3 真实只读工厂必须使用 `AkshareFetcher(enable_eastmoney_patch=False)`。

### PR167-FIX-02：延迟导入后，旧测试的 patch 路径会失效

- 现象：旧测试使用：
  - `patch("data_provider.akshare_fetcher.get_config", ...)`
- 根因：Core-M3.2 删除了 `akshare_fetcher.py` 顶层的 `get_config`，改为构造器内部延迟导入；模块级属性已经不存在。
- 结果：单独看新增测试可能正常，但完整离线测试套件会在旧测试中稳定失败。
- 正确处理：旧测试不要继续 patch 已删除的模块级属性，直接通过正式参数构造：

```python
AkshareFetcher(
    sleep_min=0,
    sleep_max=0,
    enable_eastmoney_patch=False,
)
```

- 固定规则：修改构造器依赖注入方式时，必须全仓搜索旧 patch 路径、旧构造调用和旧 fixture，不能只改新增测试。

### PR167-FIX-03：测试替身必须满足被测管理器的最小接口

- 现象：`FakeAkshareFetcher` 只有 `name`，交给 `DataFetcherManager` 后报错。
- 根因：`DataFetcherManager` 初始化时会按照 `fetcher.priority` 排序。
- 正确处理：测试替身至少补齐：

```python
class FakeAkshareFetcher:
    name = "AkshareFetcher"
    priority = 1
```

- 固定规则：创建 Fake/Stub 前先检查真实调用方在构造、排序、路由和执行阶段访问哪些属性；不能只模仿当前断言里看到的字段。

### PR167-FIX-04：不要用复杂的 `sys.modules` 父包替换验证简单构造器分支

- 现象：为了验证延迟导入，测试同时替换 `sys.modules`、`src.config`、`src.patches` 父包属性。
- 问题：Python 的模块缓存、父包属性和测试执行顺序会让这种测试不稳定；还可能意外加载真实补丁模块。
- 风险：真实 `eastmoney_patch` 模块存在 User-Agent、NID/Cookie 相关初始化逻辑，不应在离线测试中被无意导入执行。
- 正确处理：
  - 行为链路测试使用显式 `enable_eastmoney_patch=False`。
  - 构造器三态结构可使用轻量 AST 静态测试。
  - 只有确实需要验证运行时导入时，才使用局部、可控且不加载真实副作用模块的替身。
- 固定规则：优先测试公开接口和可观察结果，不为验证一条 import 语句搭建复杂模块缓存模拟。

### PR167-FIX-05：不能只盯新增测试，必须检查完整离线测试套件

- 现象：新增 `tests/test_public_market_readonly.py` 多次调整后，CI 仍在 `Offline test suite` 失败。
- 根因：真正失败来自仓库原有的 `tests/test_etf_daily_routing.py`，不是新增测试文件。
- 正确处理：当 CI 执行 `python -m pytest -m "not network"` 时，应把它视为全仓回归；修改公共构造器、模块属性或依赖注入方式后，必须搜索所有旧调用方。
- 固定规则：
  1. 先看 CI 实际执行命令。
  2. 判断是单文件测试还是全仓测试。
  3. 全仓搜索被删除或改名的符号。
  4. 再决定改业务程序还是改旧测试。

### PR167-FIX-06：不要根据 CI 步骤摘要猜根因

- 现象：GitHub 只显示 `Offline test suite` 失败，前几次在没有完整失败尾部时推测了多个原因。
- 问题：步骤级结论只能定位失败阶段，不能证明具体测试和具体异常。
- 正确处理：优先获取 workflow job logs 的失败尾部；日志接口截断时，结合：
  - CI 实际执行命令；
  - PR 改动范围；
  - 全仓旧符号搜索；
  - 失败是否可稳定复现；
  逐步收口。
- 固定规则：没有异常名、失败测试名或明确静态证据时，不把推测写成“已确认根因”。

### PR167-FIX-07：测试修复不能以削弱覆盖为代价

- 现象：为尽快让 CI 通过，曾把较完整测试压缩为更小测试集，导致部分边界覆盖丢失。
- 风险：CI 变绿不代表原安全契约仍被充分验证。
- 正确处理：最小修复只修改失败断言、替身接口或过期调用；其他既有覆盖应保留。确需删除测试时，必须说明删除原因并补上等价覆盖。
- 固定规则：修复前后对比测试函数清单，确认没有无意删除凭证隔离、非法输入、无副作用、超时和脱敏等关键测试。

### PR167-FIX-08：禁止用部分内容整体覆盖长文件

- 现象：通过 GitHub Contents API 更新长测试文件时，曾把片段误当成完整内容提交，导致文件被截断为 22 行。
- 正确处理：
  1. 写入前先获取完整文件内容和当前 blob SHA。
  2. 在完整内容上做单点替换。
  3. 写入后立即重新读取，核对总行数、文件头、文件尾和关键测试函数。
  4. 若发生截断，立即从上一提交的 blob 完整恢复，再重新做最小修改。
- 固定规则：`update_file` 是整文件替换，不是补丁追加；任何长文件写入后都必须进行完整性复核。

### PR167-FIX-09：一次修一个确定问题，不连续叠加猜测性提交

- 现象：同一个 PR 因多次猜测性测试修改产生多个修复提交，增加了回溯和判断难度。
- 正确处理：遵循 Plan → Build → Judge：
  1. Plan：先锁定 CI 命令、失败范围和受影响旧调用。
  2. Build：只改一个确定根因。
  3. Judge：核对 diff、文件完整性和新 CI，再继续下一步。
- 固定规则：同一轮 CI 未给出新证据前，不重复重写同一个测试文件。

### PR167-FIX-10：安全说明必须与真实代码路径一致

- 现象：文档曾写“不会读取其他 Provider Token/API Key 状态”，但构造器仍会进入通用 Config。
- 正确处理：先通过代码和测试证明显式 `False` 路径不调用 `get_config()`、不执行补丁，再更新文档。
- 固定规则：安全边界文案必须以可验证的调用链为依据，不能仅根据“只创建了一个 Provider”推断。

## 后续同类改动的强制检查清单

### Build F1：截图预览不能等同于自动识别或自动写入

- 风险：将“截图已选择”误标为“已识别”，或在没有人工核对时直接覆盖持仓，会把错误数据写入错误用户或错误领域。
- 固定规则：截图必须明确标注为仅本机预览、未上传；自动识别接入前只允许“按截图手动填写并确认”，最后确认才写入当前用户且当前股票/基金中心。

### Build E5：变更历史不能跨用户或变成高敏感日志

- 风险：把历史记录按全局查询，会让用户切换时看到其他人的持仓；记录截图、设置或凭证会把简单审计功能变成敏感数据存储。
- 固定规则：历史必须按 `user_id` 查询并保留股票/基金类型；只保存必要的持仓快照与操作类型，接口只允许 localhost。

### E5-FIX：分区页面不能只在前端隐藏另一领域的数据

- 风险：即使页面不展示另一类历史，只要接口返回全部数据，股票/基金分区仍不完整且容易在后续改动中泄露到错误区域。
- 固定规则：股票、基金分区必须由后端查询参数收口，页面只请求当前领域的数据。

### Build E6：SQLite 既有表扩字段前先考虑兼容存量用户

- 风险：直接为已部署的本机表添加新列会要求迁移，失败时会让已有持仓页面无法加载。
- 固定规则：轻量历史增强优先使用版本兼容的受控快照结构；旧记录必须保持可读，不能为新显示功能清空用户数据。

### 侧栏：窄宽度下中心切换不能并排挤压中文标签

- 风险：侧栏内并排放置“股票中心 / 基金中心”会把中文标签截断，且居中导航在窄窗口中不利于快速扫读。
- 固定规则：桌面左侧栏的中心切换和导航均使用靠左的纵向排列；发布前至少检查股票、基金及持仓中文标签的完整可见性。

### Build E7：当前用户选择也属于用户隔离数据

- 风险：只在前端内存保存当前用户，重启后会回到“本人”，容易让用户误以为刚才录入的股票或基金持仓丢失或属于另一个用户。
- 固定规则：当前用户必须以低敏本机偏好写入 SQLite；读取时只接受现存用户，删除当前用户时安全回退到“本人”，不修改任何其他用户持仓。

### Build E8：问答页面分开不等于会话上下文已经分开

- 风险：若股票和基金页面共用同一个会话 ID 或浏览器恢复键，切换页面后会把另一领域的追问上下文带入当前问题。
- 固定规则：会话 ID、会话列表筛选和本机恢复键必须同时按“当前用户 + 股票/基金领域”分区；历史无分区会话不得自动混入。

### Portable-M2.1：校验入口不能暗含安装语义

- 风险：把“校验更新包”接到安装版的检查/安装按钮，或在校验完成后自动解压、替换、删除、重启，会越过用户对便携数据目录的保护边界。
- 固定规则：M2.1 只读计算用户手动选择 ZIP 的 SHA-256，并明确提示“尚未安装或替换任何文件”；入口仅在便携版显示。ZIP 结构检查与真实替换更新分别后置，M2.2 必须先备份、临时解压、保留 `data/config/logs/plugins`、启动验证和失败回退，并取得用户关键授权。

### Portable-M2.1.2：哈希一致不等于可运行的便携包

- 风险：ZIP 即使与对应 `.sha256` 一致，也可能缺少中文 EXE、打包后的 backend EXE 或保留数据目录，若误判为可更新会在 M2.2 产生不可恢复的启动或数据风险。
- 固定规则：在任何更新安装前，M2.1.2 只读检查 ZIP 中央目录，必须同时确认中文 EXE、backend EXE、`data/logs/config/plugins`；缺少任一项即明确拒绝，仍不得解压、替换或修改用户文件。

### Portable-M2.2A：恢复点不能把用户数据当作程序文件

- 风险：若更新前递归备份或恢复整个便携目录，会把更新期间新写入的持仓、配置和日志倒退为旧值。
- 固定规则：恢复点只处理可替换程序文件，`data/config/logs/plugins` 永远排除在备份、替换和回退之外；恢复点必须位于程序目录内的受控 `.portable-update-backups/` 路径。

### Portable-M2.2B：替换异常必须先回退程序，不能回退用户数据

- 固定规则：程序文件替换失败时立即使用恢复点回退；data/config/logs/plugins 不参与替换或回退。

1. 公共构造器、函数签名、模块级符号发生变化时，全仓搜索旧调用和旧 patch 路径。
2. 确认 CI 是单文件测试还是完整离线测试，不能把本地单测通过等同于全仓回归通过。
3. Fake/Stub 必须覆盖真实调用方访问的最小接口，包括排序字段、名称和执行方法。
4. 真实只读路径必须显式关闭会进入通用 Config 或产生 Cookie/NID 副作用的兼容逻辑。
5. 优先使用正式参数注入，不使用复杂 `sys.modules` 技巧绕过公开接口。
6. 不加载真实联网或带副作用模块完成离线单元测试。
7. 测试修复不得无意删除既有安全覆盖。
8. GitHub `update_file` 写长文件后必须立即复核完整行数、文件头尾和关键函数。
9. 没有完整错误证据时，将判断标为“推测”，不能宣称已确认。
10. PR 修复只推送最小提交，不自动合并。
