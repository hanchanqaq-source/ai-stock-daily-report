# App-M4.2.2 前端敏感字段安全交互与服务端契约接入

App-M4.2.2 让 Web 设置页接入 App-M4.2.1 的服务端敏感字段契约。前端只表达用户意图，不读取、不显示、不保存真实凭证原文。

## keep / set / clear 交互

- `keep`：已配置敏感字段的默认状态。用户未点击修改或清除时，前端不会把该字段放入 changed items；需要显式表达时可使用 `{ key, action: "keep" }`。
- `set`：用户点击“修改”或未配置敏感字段直接输入新值后，保存时提交 `{ key, action: "set", value: "new-value" }`。
- `clear`：用户点击“清除”并在确认对话框中确认后，保存时提交 `{ key, action: "clear" }`，不携带 `value`。

## 已配置与未配置状态

- 已配置且被服务端脱敏的敏感字段显示“已配置”，密码框保持空值，不显示原始值，也不把掩码当作可编辑内容。
- 未配置的敏感字段显示“未配置”，用户可直接输入新值；保存时使用 `action=set`。
- 输入框清空只表示当前编辑值为空，不代表清除。清除必须来自明确的“清除”按钮和二次确认。

## 修改、取消和明确清除流程

1. 已配置字段初始为 keep，不算 dirty。
2. 点击“修改”后输入框从安全空值开始。
3. 输入有效新值后生成 set 意图。
4. 点击“取消修改”后回到 keep，不再算 dirty。
5. 点击“清除”会打开确认对话框；未确认不会生成 clear。
6. 确认清除后生成 clear 意图，保存 payload 不带敏感 value。
7. 保存成功或 resetDraft 后，敏感字段编辑状态重置。

## 掩码占位保护

前端会阻止以下占位内容作为新密钥提交：

- 当前服务端返回的 `maskToken`
- `masked-value`
- 连续星号占位，例如 `********`

这些值不会被自动转换成 `action=set`。

## 非敏感字段兼容性

普通非敏感字段继续使用原有 `{ key, value }` 保存语义，dirty、reset、validate、save 和分类展示逻辑保持兼容。
