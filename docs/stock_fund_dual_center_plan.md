# 股票基金双中心架构规划

## 1. 原因

当前程序以股票分析为核心，基金只在个人持仓页中作为一个分区出现。现状会造成以下问题：

- “问股票”和“问基金”没有独立的信息架构；
- 股票代码、股票策略、股票上下文可能误用于基金问题；
- 基金行业穿透、行业周期、基金对比和持仓建议没有明确入口；
- 用户切换虽然已经出现，但股票数据、基金数据和分析记录尚未形成清晰的用户隔离边界；
- 后续继续在现有股票页面旁边零散增加基金功能，容易再次漏功能或混淆功能归属。

本规划将“股票基金质量分析系统”重构为一个中文 EXE 内的两个一级工作区：

1. 股票中心；
2. 基金中心。

系统设置、用户管理、用量、日志和便携更新作为共用能力，不复制成两个程序。

## 2. 固定原则

- 项目名称保持“股票基金质量分析系统”。
- 日报/推送显示名称保持“AI股票基金每日信息报告”。
- 最终体验保持“下载 ZIP → 解压 → 双击中文 EXE → 直接使用”。
- 股票和基金的页面、问题上下文、持仓模型、分析结果必须分开。
- 当前用户在两个中心之间保持一致，但每个用户的数据必须隔离。
- 不伪造基金净值、行业持仓、基金经理、周期评分或 AI 建议。
- 数据缺失时必须显示来源、日期和缺失原因。
- 不接真实账户、自动交易或未授权的真实 Provider。
- 不读取、打印或上传密钥。
- 不降低 Electron safeStorage / Windows DPAPI 安全等级。
- 不自动发布 Release。
- 所有实现继续采用 Plan → Build → Judge。

## 4. 导航与首页

### 4.1 一级切换

左侧导航顶部提供明确的“股票中心 / 基金中心”切换。切换后只显示当前中心所属的菜单，避免股票功能和基金功能混排。

### 4.2 当前用户

当前用户选择器固定显示在两个中心的公共位置。切换用户后：

- 股票中心只读取该用户的股票持仓和股票分析记录；
- 基金中心只读取该用户的基金持仓和基金分析记录；
- 不允许使用另一个用户的持仓生成建议；
- 设置中的系统级配置不随用户重复保存。

### 4.3 启动页

首次启动显示“股票中心”和“基金中心”两个大卡片。后续启动可以记住上次进入的中心，同时保留一键切换入口。

## 6. 问答边界

### 6.1 问股票

- 接受股票代码、股票名称、股票策略和股票报告追问；
- 可以使用股票分析技能、股票历史报告和股票自选上下文；
- 不显示基金示例，不自动解释为基金问题；
- 基金代码或明确基金问题应提示前往“问基金”。

### 6.2 问基金

- 接受基金代码、基金名称、基金持仓、行业暴露、风险和配置问题；
- 不加载股票策略、自选股、股票报告追问上下文；
- 回答必须标记基金数据日期和来源；
- 没有真实基金数据时只能解释分析框架或显示“数据未接入”，不能生成虚假持仓和净值；
- 明确股票问题应提示前往“问股票”。

### 6.3 会话隔离

股票会话与基金会话使用独立的会话范围和最近会话记录。切换中心不能把上一中心的活动上下文带入当前问题。

## 7. 用户与数据模型边界

```text
用户
├─ 股票域
│  ├─ 股票持仓
│  ├─ 证券账户
│  ├─ 股票分析记录
│  └─ 股票告警
└─ 基金域
   ├─ 基金持仓
   ├─ 基金观察列表
   ├─ 基金分析记录
   └─ 基金配置建议
```

系统级数据单独保存：

- 数据源配置；
- AI/模型配置；
- 通知配置；
- 安全凭证；
- 版本与更新状态。

任何用户删除操作必须明确目标用户，不得同时删除系统级设置或其他用户数据。

## 8. 基金中心功能规划

### 8.1 基金资料

基金基础资料至少包含：

- 基金代码与名称；
- 基金类型；
- 基金经理；
- 规模与成立日期；
- 净值日期；
- 数据来源与抓取时间。

### 8.2 基金持仓与行业穿透

持仓快照至少包含：

- 报告期；
- 前十大持仓；
- 单项权重；
- 行业映射；
- 已披露持仓合计；
- 未披露部分提示。

行业穿透结果不能把未知行业强行归类，也不能把相邻赛道当作目标赛道。

### 8.3 行业周期与景气度

行业周期模块按“证据 → 判断 → 对基金影响”输出：

1. 行业价格、订单、资本开支、库存、政策或盈利证据；
2. 数据日期和来源；
3. 周期阶段：复苏、扩张、过热、放缓或收缩；
4. 置信度和缺失项；
5. 对基金当前持仓暴露的影响。

“生产力/产业趋势”作为独立证据维度，关注技术进步、产能效率、渗透率和长期需求，不等同于短期涨跌预测。

### 8.4 基金对比

对比维度至少包括：

- 基金类型与策略；
- 行业集中度；
- 前十大持仓集中度；
- 规模和流动性；
- 风险与回撤；
- 与当前用户持仓的重复度；
- 数据日期与来源。

### 8.5 仓位、风险与调仓建议

建议必须：

- 只基于当前用户的基金持仓；
- 区分事实、推断和建议；
- 标记数据是否过期；
- 说明行业集中、基金重叠、风格偏移和单基金风险；
- 不执行交易；
- 不声称保证收益；
- 在没有真实数据时不生成具体买卖结论。

## 9. GitHub 项目与能力适配

已经引入或参考的 GitHub 项目只作为底层能力来源，通过适配层接入，不直接决定用户界面。

每个适配器必须检查：

- 许可证与允许的使用方式；
- 数据范围是股票、基金还是通用能力；
- 网络请求、缓存和失败行为；
- 是否读取环境变量或敏感配置；
- 是否会连接真实账户或执行交易；
- 输出是否带来源和时间；
- 依赖是否适合 PyInstaller 和 Windows Portable。

股票适配器不能直接用于基金结论；基金适配器也不能修改股票现有契约。

## 10. 需求追踪与阶段关闭门禁

为防止“规划中写了、程序中遗漏”，从 Build A 开始维护需求追踪矩阵。每条用户要求必须拥有唯一编号，并完整映射到页面、数据、实现和验收证据。

| 字段 | 必填内容 |
|---|---|
| 需求编号 | 稳定且可追踪，例如 `DUAL-001` |
| 用户原始要求 | 保留用户确认过的简明表述 |
| 所属区域 | 股票中心、基金中心、用户或共用能力 |
| 页面与路由 | 可见入口和目标路由 |
| 数据模型/API | 对应存储、接口和隔离边界 |
| 实现 PR | 实际完成该需求的 Draft PR |
| 自动测试 | 单元、集成或端到端测试 |
| Windows 证据 | Portable 实机或 CI 截图、日志 |
| 状态 | 未开始、实现中、已验证或延期并说明原因 |

阶段关闭门禁：

- 没有页面入口、数据模型、实现 PR、测试和 Windows 证据映射的需求，不得标记完成；
- 用户新要求先加入矩阵，再进入 Build；
- 每个 Build PR 的描述必须列出“已覆盖需求”和“明确未包含内容”；
- Judge 必须逐项对照矩阵，不能只依据代码存在或 CI 绿色；
- 延期项必须记录原因和后续阶段，不允许静默遗漏；
- 股票中心和基金中心分别核对，不能用一个中心的实现代替另一个中心。

## 11. 用户数据持久化与数据库迁移

当前用户、快速持仓和分析记录不能长期停留在内存状态。持久化至少覆盖：

- 用户资料和当前用户选择；
- 每个用户的股票持仓、股票观察项、股票分析记录；
- 每个用户的基金持仓、基金观察项、基金分析记录；
- 股票问答与基金问答的独立会话索引；
- 上次进入的中心和非敏感界面偏好。

数据库要求：

- 所有业务表必须包含明确的用户归属和股票/基金领域归属；
- 建立 `schema_version`，每次结构变化使用可审计迁移；
- 升级前备份用户数据，迁移使用事务，失败时保持旧数据可用；
- 为迁移提供回滚或兼容读取方案，不覆盖、不清空已有持仓；
- 删除用户必须二次确认并限定该用户数据，不能删除其他用户或系统设置；
- 自动测试覆盖重启后数据仍存在、用户之间不串数据、股票基金不串域；
- Portable-M2 更新必须验证旧版本数据升级、失败回退和再次启动。

## 12. 数据来源、时效与缺失状态

股票和基金的事实数据统一携带以下元数据：

- 数据来源；
- 抓取时间 `fetched_at`；
- 数据生效日或报告期 `effective_at/report_period`；
- 是否过期 `stale`；
- 可选置信度；
- 缺失字段和缺失原因。

基金持仓通常来自定期披露，界面必须区分“最新净值日期”和“最新持仓报告期”，不能把旧持仓显示为实时持仓。来源失败时保留上次成功数据并明确标记过期，不得静默替换成 Mock 数据。测试 fixture 必须带“测试数据”标识，正式便携版不得把 fixture 当真实结论。

## 13. AI 输出、行业证据与日报分区

AI 输出统一拆分为：

1. 可核对事实；
2. 基于事实的推断；
3. 风险提示；
4. 非自动执行的建议。

基金行业周期、景气度和生产力/产业趋势必须展示证据、日期、阶段、置信度及缺失项；短期周期判断与长期产业趋势分开，不能用长期趋势直接得出短期买卖结论。

日报/推送显示名称继续固定为“AI股票基金每日信息报告”，内容按当前用户分区：

- 股票部分：股票持仓、关注项、风险和报告摘要；
- 基金部分：基金持仓、行业暴露、周期证据、重叠度和风险摘要；
- 共用部分：数据日期、来源、缺失项和当前用户；
- 某一中心无数据时显示“暂无数据/尚未接入”，不伪造内容。

## 14. 配置、密钥与发布更新边界

- 普通配置与安全凭证分离保存；
- 安全凭证继续使用 Electron safeStorage / Windows DPAPI，不写入明文便携文件；
- `.env` 只能作为无真实密钥的模板或开发兼容入口，不随正式 ZIP 分发真实凭证；
- 日志、诊断包和错误提示不得打印 token、webhook、API key 或完整敏感配置；
- 不接真实账户、自动交易或未经授权的 Provider；
- 不自动发布 GitHub Release；
- 版本号、ZIP、SHA-256、更新清单和回滚包必须来自同一次已验收构建；
- 一键更新在 Portable-M2 单独规划和实现，更新前备份用户数据，校验 SHA-256，失败后恢复旧版本；
- 未经用户授权，不改变 PR Draft 状态、不合并、不发布。

## 15. Windows 回归矩阵与后续清理

每个影响桌面端、数据目录或后端启动的 Build，都必须至少验证：

| 场景 | 必须结果 |
|---|---|
| Portable 首次启动 | 双击中文 EXE 可直接进入，不要求开发环境 |
| 后端启动 | 使用 packaged backend，仅监听 `127.0.0.1` |
| 控制台体验 | 正常启动不弹出后端黑框 |
| 双中心导航 | 股票与基金入口、标题、空状态明确分开 |
| 用户切换 | 两个中心均切换到对应用户数据 |
| 数据重启 | 关闭并重新启动后用户与持仓仍存在 |
| 旧数据升级 | 旧版本数据迁移成功且不丢失 |
| 迁移失败 | 可回退旧数据和旧版本 |
| 问答隔离 | 股票问题和基金问题不串会话与上下文 |
| 日报 | 当前用户的股票、基金内容正确分区 |
| 安全日志 | 不含密钥，不进入 development Python 模式 |
| 构建产物 | ZIP、中文 EXE、内置后端和 SHA-256 完整 |

架构稳定后再处理非阻断清理项：

- Windows cp1252 日志编码警告；
- ZIP 体积、启动时间和内存占用；
- 重复依赖与无用资源；
- 更清晰的离线、超时和数据过期提示。

这些清理不得通过跳过测试、弱化断言或降低安全等级实现。

## 16. Build 阶段拆分

### Build A：双中心骨架

- 一级中心切换；
- 新路由与旧路由兼容；
- 股票功能迁移到股票中心；
- 基金页面骨架；
- 全局用户选择器；
- 不接真实基金数据。

### Build B：持仓分域

- 将现有混合持仓页拆为股票持仓和基金持仓；
- 保留用户快速录入；
- 股票、基金状态分别保存；
- 切换用户后严格隔离。

### Build C：基金数据契约

- 基金资料、净值、持仓快照、行业映射和来源字段；
- Mock/fixture 只用于测试并明确标识；
- Provider 接口与真实实现分离。

### Build D：基金分析能力

- 基金对比；
- 行业穿透；
- 行业周期与景气度；
- 产业趋势/生产力证据；
- 风险和仓位建议。

### Build E：持久化与便携验收

- 用户、当前用户、股票/基金持仓和会话索引持久化；
- schema 版本、升级前备份、事务迁移和回滚；
- 旧版本数据升级以及跨用户、跨领域隔离测试；
- Windows Portable 重新构建；
- 真实启动、重启持久化和升级失败回退验收。

Portable-M2 安全一键更新在双中心结构稳定后继续，避免更新器与目录结构同时大改。

## 17. Judge 验收清单

### 需求闭环

- [ ] 每条已确认要求都进入需求追踪矩阵。
- [ ] 每条完成项都映射页面、数据、实现 PR、自动测试和 Windows 证据。
- [ ] 延期项有明确原因和后续阶段，没有静默遗漏。

### 架构

- [ ] 一个中文 EXE 内存在股票中心和基金中心。
- [ ] 两个中心的导航、标题、输入提示和空状态明确分开。
- [ ] 共用设置不重复。
- [ ] 旧路由安全跳转。

### 用户隔离

- [ ] 切换用户后股票持仓正确切换。
- [ ] 切换用户后基金持仓正确切换。
- [ ] 股票和基金会话上下文互不串用。
- [ ] 不读取其他用户的持仓生成建议。

### 持久化与迁移

- [ ] 关闭并重启后当前用户、股票持仓和基金持仓仍存在。
- [ ] 数据库含 schema 版本，升级前备份，迁移失败可回退。
- [ ] 旧版本升级不清空、不覆盖、不串用用户数据。
- [ ] 股票域与基金域数据隔离。

### 基金能力

- [ ] 基金资料包含来源、抓取时间、生效日/报告期和过期状态。
- [ ] 行业穿透保留未知项。
- [ ] 行业周期输出证据、阶段、置信度和缺失项。
- [ ] 基金对比说明持仓重复和集中度。
- [ ] 建议区分事实、推断、风险提示和建议。
- [ ] 行业周期与长期生产力/产业趋势不混为短期买卖结论。
- [ ] 无真实数据时不伪造结果。

### 日报

- [ ] 名称保持“AI股票基金每日信息报告”。
- [ ] 按当前用户分为股票、基金和共用数据说明。
- [ ] 无数据或数据过期时明确显示，不伪造内容。

### 安全与便携

- [ ] Web 仍只绑定 `127.0.0.1`。
- [ ] 不读取或打印密钥。
- [ ] 不接真实账户和自动交易。
- [ ] safeStorage / Windows DPAPI 不降级。
- [ ] Windows Portable ZIP、中文 EXE、内置后端和 SHA-256 继续通过。
- [ ] packaged backend、`127.0.0.1`、主页面加载和非 development Python 均有日志证据。
- [ ] 正式 ZIP 不携带真实密钥，日志与 diagnostics 不泄露安全凭证。
- [ ] 不自动发布 Release。

## 18. 回滚

每个 Build 使用独立 Draft PR。若某阶段失败，只回滚该阶段新增的路由、页面、状态和适配器，不回退已稳定的 Windows Portable-M1，也不破坏现有股票功能。
