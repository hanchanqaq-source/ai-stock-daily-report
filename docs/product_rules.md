# AI 股票基金日报系统产品规则总纲
全球市场指数矩阵与页面切换模型见 `docs/market_index_matrix.md`。官方指数和系统计算指标必须分开展示，系统计算指标必须标注“非官方指数”。

资产实时行情能力标记见 `docs/quote_capability.md`。股票 / ETF 可展示实时或接近实时行情；场外基金只能展示净值或估算净值，不能称为实时涨跌。
本文档记录项目级产品规则、数据分层规则、安全规则和后续优化路线。后续 Codex / AI 开发在涉及账户、资产、报告、数据保存、本地工作台或清理能力前，必须先参考本文档，并保持“公共市场日报 + 本地数据主权 + 不做复杂实名用户系统”的产品边界。

资产自动补全框架见 `docs/asset_enrichment.md`，所有补全结果必须保留来源、状态和置信度。
统一资产模型见 `docs/asset_model.md`，后续账户分组、个人雷达、网页工作台均应复用该模型。资产状态管理见 `docs/asset_status_manager.md`，所有清仓、归档、删除操作必须先预览再确认。持有 vs 收藏对比分析见 `docs/holding_watch_compare.md`。账户动态页面模型见 `docs/account_page_model.md`。代码自动识别框架见 `docs/code_identifier.md`，所有识别结果必须遵守信息来源可查证规则。

## 1. 项目定位

本项目不是单纯日报脚本，正在逐步升级为：

- 公共股票 / 基金市场日报系统
- 历史数据分析系统
- 账户 / 分组观察系统
- 未来本地网页工作台
- 未来个人股票 / 基金 / 企业观察中心

当前阶段仍以 GitHub Actions + Discord + 公共市场数据为主。默认优先保证公共日报、周报、历史快照、公共市场观察与安全 dry-run 能力稳定，不应把当前阶段扩展成复杂实名用户系统、资金记账系统或在线交易系统。

## 2. 账户 / 分组规则

账户 / 分组用于表达观察分组、投资组合或主题组合，不用于表达实名身份。

1. 账户不是实名用户。
2. 账户只是观察分组 / 投资组合。
3. 不需要邮箱。
4. 不需要手机号。
5. 不需要身份证。
6. 不需要登录密码。
7. 账户可以叫：
   - 张三组
   - 张四组
   - 我的基金组
   - AI硬件组
   - 稳健基金组
8. 每个账户下面是 `assets` 资产列表。
9. 有基金就显示基金相关页面。
10. 有股票就显示股票相关页面。
11. 基金和股票都有，就显示两个页面。
12. 没有资产，就显示“请添加资产”。

## 3. 统一资产模型规则

所有被观察、持有、收藏、分析或归档的对象统一叫 `asset`。

资产类型包括：

- `fund`：基金
- `stock`：股票
- `etf`：ETF
- `company`：企业
- `industry`：行业
- `theme`：主题
- `index`：指数
- `unknown`：暂未确认

资产基础字段包括：

- `asset_id`
- `type`
- `code`
- `name`
- `market`
- `tags`
- `status`
- `weight_level`
- `source_status`

其中 `status` 包括：

- `holding`：持有中
- `watching`：收藏 / 观察
- `cleared`：已清仓
- `archived`：已归档
- `deleted`：真正删除，后续谨慎使用

统一资产模型的目标是让股票、基金、企业、行业、主题和指数共享相同的观察、标签、来源校验、状态管理与报告输入规则，避免为不同对象新增互相割裂的平行实现。

## 4. 持有 / 收藏 / 清仓 / 删除规则

1. 持有中的资产显示在持有列表。
2. 收藏 / 观察中的资产显示在收藏列表。
3. 已清仓资产不再显示为当前持有。
4. 已清仓资产可以保留历史记录。
5. 归档资产默认不显示在主页面。
6. 删除资产属于危险操作，必须先预览再确认。
7. 后续网页工作台必须支持：
   - 设为持有
   - 设为收藏
   - 设为已清仓
   - 恢复观察
   - 归档
   - 删除
   - 编辑标签

清仓、归档和删除都不应被静默触发。清仓偏向状态变化，归档偏向从主界面隐藏，删除代表真正移除，必须谨慎使用。

## 5. 持有 vs 收藏对比规则

后续系统要支持对比：

- 我买了的方向表现如何
- 我收藏但没买的方向表现如何
- 收藏方向是否比持有方向更强
- 持有方向是否出现风险
- 收藏方向是否只是短线热

输出只能写观察，不能写交易建议。

禁止出现：

- 买入
- 卖出
- 加仓
- 减仓
- 必须买
- 必须卖

允许表达的方向包括风险提示、趋势观察、数据质量说明、关注点变化和需要继续确认的信息，但不得将这些观察包装成直接交易指令。

## 6. 信息来源可查证规则

信息来源可查证规则以 `docs/source_verification.md` 为准。所有资产识别、标签补全、基金持仓判断、企业归属判断、行业 / 概念归类和正式分析结论都必须遵守以下规则：

1. 能查到，才写成确定信息。
2. 查不到，必须标记 `unknown`。
3. 不确定，必须标记 `pending_confirmation`。
4. 多来源冲突，必须标记 `conflict`。
5. 不能编造股票名称。
6. 不能编造基金名称。
7. 不能编造企业名称。
8. 不能编造行业标签。
9. 不能编造概念标签。
10. 不能编造基金持仓。
11. 不能编造数据来源。
12. 待确认信息不能进入正式分析结论。
13. 用户必须可以手动修改系统建议标签。

## 7. 代码自动识别规则

后续 P5-F / P5-G 会实现代码识别和自动补全，但必须遵守信息来源可查证规则。

预期支持：

- `257070` → 基金
- `002463` → A股股票
- `513100` → ETF / 基金
- `00700.HK` → 港股
- `AAPL` → 美股

上述示例只代表预期识别方向，不能作为无来源推断依据，系统不能乱猜。

规则：

1. 先查可验证来源。
2. 基金和股票都可能时，必须让用户确认。
3. 查不到时标记 `unknown`。
4. 不确定时标记 `pending_confirmation`。
5. 冲突时标记 `conflict`。
6. 自动补全名称、类型、市场、标签时必须带来源和置信度。

## 8. 数据分层规则

DATA-R1：公共数据轻量化保存规则。

GitHub 适合保存：

- 代码
- 公共行情摘要
- A股每日市场快照
- 指数数据
- 上涨 / 下跌家数
- 成交额
- 涨跌停差
- 行业 / 概念强弱
- 趋势分析结果
- 月度归档摘要
- 公共日报 / 周报
- example 配置

GitHub 不适合保存：

- 完整大数据库
- 分钟级行情
- 大量图片
- 大量日志
- 真实账户配置
- 真实金额
- 成本价
- 账户资产
- Token
- Webhook
- API Key

公共仓库默认只保存轻量、公共、可重建、可审查的数据。任何会暴露个人资金、真实账户、密钥或长期不可重建私人数据的内容，都不应进入 public 仓库。

## 9. 本地数据主权规则

GLOBAL-R2：本地数据主权与可迁移规则。

真正重要、长期积累、有个人价值的数据，优先留在本地。

本地适合保存：

- 真实账户 / 分组配置
- 真实基金代码 / 股票代码
- 持有 / 收藏 / 已清仓状态
- 真实金额
- 成本价
- 用户自己修正过的标签
- 个人分析结果
- 本地 SQLite 数据库
- 备份包

后续本地网页工作台必须支持：

- 导出全部本地数据
- 导出账户配置
- 导出资产列表
- 导出持有 / 收藏 / 已清仓记录
- 导出 SQLite / JSON / CSV
- 导入旧备份
- 生成备份包
- 迁移到新电脑

本地数据应可备份、可恢复、可迁移，不应被绑定到某一次 GitHub Actions 运行、某个 webhook、某台机器的不可读缓存或无法导出的内部格式。

## 10. public / private / 本地三层规则

- **GitHub public**：只放公共、轻量、可重建的数据和 example 配置。
- **GitHub private**：可以放低敏感配置和非敏感分析结果，但仍不建议放 Token、Webhook、API Key、完整资金明细。
- **本地网页工作台**：保存真实账户数据、真实金额、成本价、个人分析结果、长期备份。

三层边界的默认判断原则是：越私人、越长期、越难重建、越接近真实资金或密钥的数据，越应该留在本地；越公共、越轻量、越可重建的数据，才适合放入 GitHub。

## 11. 危险操作确认规则

所有危险操作必须遵守：

先预览 → 说明影响 → 用户确认 → 再执行。

适用范围：

- 删除资产
- 清仓资产
- 归档资产
- 清理缓存
- 清理日志
- 清理图片
- 清理 artifact
- 批量导入资产
- 批量修改标签
- 推送私人报告
- 写入真实配置

默认不静默删除，不静默推送。任何自动化能力如果可能改变用户资产状态、清理历史记录、覆盖配置、发送私人内容或写入真实数据，都必须提供可理解的预览和确认流程。

## 12. 清理中心规则

后续 Web-P12 清理中心必须遵守：

1. 清理功能集中放在“清理中心”页面。
2. 默认只扫描，不删除。
3. 先告诉用户哪些能清。
4. 说明为什么清。
5. 说明清理后有什么影响。
6. 用户勾选范围。
7. 用户确认后才执行。
8. 核心历史数据、归档摘要、私人配置、密钥默认禁止清理。

清理中心的目标是帮助用户理解存储占用和安全边界，而不是把删除能力隐藏在后台任务、自动修复或默认优化流程中。

## 13. 报告输出规则

1. 报告只写观察，不写直接交易建议。
2. 可以写风险、趋势、数据质量、观察重点。
3. 不写买入、卖出、加仓、减仓。
4. 数据不足时必须说明。
5. 来源不确定时必须标记待确认。
6. 不能把推测写成事实。
7. 不能把 example 数据写成真实数据。

报告面向辅助观察和复盘，不替代用户判断，不应输出指令式交易结论。正式报告、demo 报告、个人雷达和后续网页工作台都必须清楚区分公共数据、example 数据、待确认信息和真实个人数据。
