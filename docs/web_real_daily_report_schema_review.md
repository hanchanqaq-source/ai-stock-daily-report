# Web-P38 真实日报接入前 schema 高规格复核文档

## 1. 阶段定位

Web-P38 是真实日报接入前的 schema 高规格复核阶段，只用于在设计层面复核未来真实日报 payload 是否可以安全、稳定、可回滚地映射到当前 Web mock-only 日报展示链路。

Web-P38 不是以下任何接入阶段：

- 不是真实 provider 接入阶段。
- 不是真实 API 接入阶段。
- 不是 AI / Agent 接入阶段。
- 不是通知接入阶段。
- 不是账户接入阶段。
- 不是数据库接入阶段。
- 不是交易接入阶段。

当前仍保持 mock-only，不读取真实日报、不新增真实日报 JSON 示例、不修改 TypeScript 运行逻辑、不修改 fixture / adapter / guard、不改变页面视觉设计。

当前项目名固定为“股票基金质量分析系统”。

当前日报名固定为“AI股票基金每日信息报告”。

## 2. 当前基线

当前 Web mock-only 日报链路固定为：

```text
mock fixture → adapter → DailyReportViewModel → guard → preview model/tests
```

该链路的含义如下：

| 环节 | 当前职责 | Web-P38 边界 |
| --- | --- | --- |
| mock fixture | 提供静态脱敏 mock-only 日报数据。 | 不替换为真实日报，不新增真实 JSON 示例，不写入真实金额、真实收益率、真实持仓或凭证。 |
| adapter | 将 mock fixture 同步映射为 `DailyReportViewModel`。 | 不接 provider，不读取网络、文件、环境变量或真实 API 响应。 |
| `DailyReportViewModel` | 作为页面预览消费的稳定展示模型。 | Web-P38 只复核未来真实 schema 是否可映射，不改变模型运行逻辑。 |
| guard | 在测试中守住固定标题、数据来源、安全标签、section 顺序和敏感信息边界。 | 不作为真实 schema 校验器上线，不绕过 guard 进入页面。 |
| preview model/tests | 继续验证 mock-only 页面预览链路和网络边界。 | 不启动 dev server，不打开浏览器，不请求真实网络。 |

## 3. 未来真实日报 schema 复核清单

未来真实日报 schema 在进入任何 provider / API / 页面接入前，至少需要逐项复核以下内容：

- 字段含义：每个字段必须有清晰业务语义，避免同名字段在 provider、API、adapter、页面间含义漂移。
- 字段必填性：明确必填、可选、条件必填字段，并说明缺失时是否阻断展示。
- 默认值：明确默认值只用于展示兜底还是业务语义兜底，禁止用默认值掩盖真实数据缺失。
- 空值处理：区分 `null`、空字符串、空数组、字段不存在和 provider 不支持。
- 错误态：区分请求失败、鉴权失败、schema 校验失败、数据过期、部分数据不可用和下游渲染不可用。
- 时间格式：统一时区、日期格式、生成时间、数据刷新时间和展示时间，避免本地时间与服务端时间混用。
- 金额格式：明确币种、精度、千分位、脱敏规则和是否允许展示真实金额。
- 收益率格式：明确百分号、正负号、小数位、区间口径和是否允许展示真实收益率。
- 风险等级：明确枚举值、排序、展示文案、颜色语义和未知风险等级处理方式。
- sections 顺序：真实日报必须稳定映射到当前 sections 顺序；新增 section 需要先评估兼容性。
- 数据来源标签：必须标明数据来源、刷新状态和是否为脱敏 / mock-only / dry-run / real-readonly。
- 生成时间：必须能区分报告生成时间、数据采集时间和通知发送时间。
- 发送状态：必须区分未发送、dry-run、待人工确认、发送成功、发送失败和部分失败。
- 脱敏标签：涉及账户、持仓、金额、手机号、邮箱、Webhook、API 凭据或用户标识时必须有脱敏标签。
- 安全标签：必须标明只读、非投资建议、不交易、不自动调仓、失败隔离和回滚状态。
- 是否可稳定映射到 `DailyReportViewModel`：真实 schema 只有在字段语义、必填性、空值、错误态和顺序都可稳定映射后，才允许进入后续设计阶段。

## 4. provider 接入前置条件

未来 provider 接入前必须先满足以下条件；未满足时不得让 provider 数据进入页面展示链路：

- 数据源授权：确认数据源许可、使用范围、访问频率和展示限制。
- 请求范围：明确只请求生成日报所需的最小字段集合，不扩大到无关账户、交易或隐私字段。
- 超时：为每个 provider 设置明确超时，避免单一数据源拖垮日报生成。
- 重试：重试必须有次数、退避策略和错误分类，不得无限重试。
- 缓存：明确缓存范围、过期时间、失效策略和是否允许缓存脱敏后的结果。
- 降级：单一 provider 不可用时必须可降级，不得展示半真半假的混合数据。
- 日志脱敏：日志只记录低敏摘要、错误类别和追踪 ID，不记录 token、Webhook、API Key、真实账户标识或完整响应。
- 错误隔离：provider 失败不得影响 mock-only 本地预览，也不得拖垮无关模块。
- schema 校验失败时回退 mock-only：真实 provider payload 未通过 schema 校验时，必须回退 mock-only 本地预览，不得继续进入页面展示链路。
- provider 不得直接喂给页面：provider 输出必须先经过 schema 校验、脱敏、错误隔离和 adapter 映射，不得被页面直接消费。

## 5. 凭证与密钥管理前置条件

未来真实日报接入前必须先建立凭证与密钥管理方案：

- `.env`：只作为本地或部署环境的配置入口，不得被页面、fixture、测试快照或文档示例读取或复制。
- token：token 必须按最小权限配置，不得出现在日志、fixture、文档示例、测试快照或 PR 描述中。
- webhook：Webhook 地址必须视为敏感信息，不得打印、提交或写入 mock-only 数据。
- API 凭据：API Key、Secret、访问令牌和刷新令牌必须由环境或密钥管理系统注入。
- 本地 / CI / 部署环境差异：必须明确不同环境的凭证来源、缺省行为和禁用真实接入的方式。
- 最小权限：凭证只授予只读、必要字段和必要请求范围，不授予交易或账户修改权限。
- 禁止打印密钥：日志、错误栈、调试输出和测试失败信息不得包含任何密钥或完整凭证。
- 禁止上传密钥：不得将 `.env`、token、Webhook、API 凭据或真实账户导出文件提交到仓库。
- 禁止把凭证写入 fixture、文档示例或测试快照：所有示例必须使用明显虚构占位符或直接省略敏感字段。

## 6. 通知 dry-run 前置条件

未来通知能力进入 dry-run 前，必须先满足以下条件：

- 不真实发送：dry-run 只生成发送计划、低敏摘要和状态，不调用真实通知渠道。
- 不暴露手机号、邮箱、Webhook 地址：日志、页面、测试输出和文档示例均不得出现真实目标地址。
- 失败不影响日报生成：通知 dry-run 失败必须与日报生成隔离，不能导致日报主体不可生成。
- 日志不泄露目标地址：只允许记录渠道类型、脱敏目标标识、错误类别和追踪 ID。
- 人工确认后才能进入真实发送阶段：真实发送必须作为独立阶段，经人工确认授权、回滚方案和验收边界后才能启用。

## 7. 真实账户和持仓边界

未来涉及真实账户或持仓数据前，必须先确认以下边界：

- 用户确认：必须有明确用户授权和使用范围确认。
- 只读：只允许读取生成日报所需的最小字段，不允许修改账户或持仓。
- 字段最小化：不得读取与日报展示无关的账户明细、交易流水、身份信息或联系方式。
- 不自动调仓：系统不得根据日报结果自动调整仓位。
- 不交易：不得下单、撤单、改价、申赎或触发任何交易动作。
- 不保存原始账户数据到 mock fixture：mock-only fixture 只能保存脱敏、虚构、静态数据。
- 不把真实金额写进 mock-only 测试：测试不得包含真实资产规模、真实成本、真实收益率或真实持仓明细。

## 8. 回滚方案

真实日报后续阶段必须保持清晰回滚路径：

- 真实日报失败时回退 mock-only 本地预览：provider、API、schema、通知或账户链路失败时，页面仍可使用 mock-only 本地预览作为安全展示和排障入口。
- 不展示半真半假的数据：如果真实日报只部分成功，必须清晰标记失败并阻断混合展示，不得把真实片段和 mock 片段拼成看似完整的日报。
- schema 校验失败时不得继续进入页面展示链路：校验失败必须在页面前阻断，并保留低敏错误态、追踪 ID 和回退说明。

## 9. Web-P38 复核结论

Web-P38 的交付物仅是 schema 高规格复核文档和文档索引 / 后续阶段说明更新。完成 Web-P38 不代表真实日报 schema 已定稿，也不代表 provider、API、AI、通知、账户、数据库或交易链路可以开始接入。

后续若进入真实日报设计或 dry-run 阶段，应先基于本文档逐项补齐 schema 契约、凭证管理、provider 隔离、通知 dry-run、账户只读授权、日志脱敏、测试证据和回滚方案。
